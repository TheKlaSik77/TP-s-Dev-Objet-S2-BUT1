TP semaine 2


Conception objet et programmation d’une application simple :

Comptes en banque

 

Enoncé du problème :

Vous devez programmer la gestion d’une banque de détail : la banque gère des comptes en banque qui appartiennent à des clients (tout cela bien sûr de manière très simplifiée...).

Chaque client est repéré par son nom (pour simplifier on supposera que tous les clients ont des noms différents) et son adresse (on ne cherchera pas à décomposer l’adresse qui sera réduite à un simple type String).
Par exemple : « Tintin » habite « Moulinsart »

L’adresse d’un client peut être modifiée, mais pas son nom.

Chaque compte appartient à un client (le titulaire du compte) et est repéré par un numéro de compte (utilisez pour cela le type int). Un compte peut être crédité ou bien débité d’un certain montant (son solde étant alors modifié).
Par exemple : tintin est titulaire du compte n°1234 dont le solde est 100 euros.

Dans ce TP on ne s’intéresse pas à limiter le découvert des comptes : on peut toujours retirer tout l’argent qu’on veut même si le solde est négatif (ce n’est certes pas très réaliste mais c’est plus simple à programmer... et pour le client c’est bien pratique !).
Par exemple si le solde du compte de tintin est 100 euros et qu’il demande à débIter 1 million d’euros, pas de problème : la banque accepte et le solde du compte passe à - 999 900 euros.

Rappel : tous les attributs doivent être private.

Notion de getter : Un getter (en français : accesseur) est une méthode qui retourne la valeur d’un attribut. Le nom de cette méthode est habituellement composé du mot get et du nom de l’attribut. Par exemple dans la classe Artiste la méthode getNom :

 public String getNom() {

  return this.nom;

 }

Notion de setter : un setter (en français : mutateur) est une méthode qui permet de donner une nouvelle valeur à un attribut. Le nom de cette méthode est habituellement composé du mot set et du nom de l’attribut. Par exemple dans la classe Artiste la méthode attribuerNote pourrait s’appeler setNote:

 public void setNote(int note) {

  this.note = note;

 }

Attention : dans une classe on ne programme que les getters et setters dont on a besoin (ne pas programmer systématiquement tous les getters et setters). Par exemple, puisqu’il est écrit dans cet énoncé que le nom du client n’est pas modifiable, il ne faut pas programmer de méthode setNom dans la classe Client.

1. (fondamental)

Programmez le classe Client.
Dans le main d’une classe Essai :
- créez un client puis affichez ses caractéristiques (c’est à dire son nom et son adresse) ;
- modifiez son adresse puis affichez à nouveau ses caractéristiques.

 

2. (fondamental)

Programmez le classe Compte.
Dans le main de votre classe Essai :
- créez un compte ;
- affichez les caractéristiques de ce compte (c’est à dire le numéro de compte, les caractéristiques du titulaire et le solde) ;
- créez un autre client et encore deux autres comptes ;
- réalisez quelques opérations de crédit et de débit en affichant ensuite à chaque fois les caractéristiques du compte concerné pour voir si l’opération a bien été réalisée.


Nettoyez votre code.
Tous les noms (noms de classes, d’attributs, de méthodes, de paramètres, de variables...) doivent être soigneusement choisis de manière à rendre la lecture du code limpide.

 

3. (avancé)

Le type double (et tous les types dits à virgule flottante) est un mauvais choix pour gérer des quantités pour lesquelles les résultats des calculs doivent toujours être d’une précision absolue.

Essayez par exemple le calcul suivant :

  double solde = 1.13; // le solde contient 1 euros 13

  solde = solde - 0.22; // on débite 22 centimes, il devrait rester 0 euros et 91 centimes sur le compte

  System.out.println(solde); // Zut ! le solde est faux !! Le client ne va pas être content d’avoir perdu de l’argent.

Consultez la page http://docs.oracle.com/javase/tutorial/java/nutsandbolts/datatypes.html et réfléchissez aux raisons pour lesquelles il est à écrit à propos des types float et double : « This data type should never be used for precise values, such as currency. »

Utilisez le type long pour toutes les valeurs monétaires de votre programme. Vous les gérerez en centimes (par exemple : si il y a 10020 sur un compte cela signifie 100 euros et 20 centimes), et demandez-vous quel est le solde maximum possible pour un compte.

 

4. (fondamental)

La banque gère une liste de clients et une liste comptes.

On veut pouvoir ajouter un client, ajouter un compte, récupérer une version affichable de la liste des clients, récupérer une version affichable de la liste des comptes, récupérer le nombre de comptes, récupérer le nombre de clients, récupérer une version affichable de l’état d’un compte dont on connaît le numéro, réaliser un crédit / débit sur un compte dont on connaît le numéro, modifier l’adresse d’un client dont on connaît le nom.

Programmez la classe Banque.

Dans le main de votre classe Essai, créez la banque « Le Crédit Montreuillois », ajoutez quelques clients, quelques comptes, et essayez les fonctionnalités demandées.

 

5 (fondamental)

Un compte est à dit « à découvert » si son solde est négatif.
Ajoutez une fonctionnalité à la banque permettant de récupérer une version affichable de la liste des comptes à découvert.

 

6 (fondamental)

Dessinez le diagramme de classe de votre programme.

 

7 (avancé)

Générez automatiquement les numéros de compte. Plutôt que de saisir le numéro de compte à chaque création de compte, faites en sorte que le premier compte créé ait automatiquement le numéro 1, le second le numéro 2, etc.

 

8 (consolidation)

Développez un menu permettant à un employé de banque de réaliser des opérations sur la banque.
